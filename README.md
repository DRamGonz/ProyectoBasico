# API de Gesti√≥n de Proyectos y Tareas (Laravel 12)

## üß© Enunciado del Ejercicio

### üéØ Objetivo

Desarrollar una API REST en Laravel para gestionar **usuarios, proyectos y tareas**, implementando **roles b√°sicos**, utilizando una **capa de servicios** para manejar la l√≥gica de negocio y acceso a la base de datos.

---

### üìò Requisitos funcionales

#### 1Ô∏è‚É£ Entidades principales

**Usuarios (`users`)**

* Campos: `name`, `email`, `password`, `role` (`admin`, `project_manager`, `developer`)
* Relaciones:

  * Un `project_manager` puede tener muchos proyectos.
  * Un `developer` puede tener muchas tareas asignadas.

**Proyectos (`projects`)**

* Campos: `name`, `description`, `start_date`, `end_date`, `user_id` (responsable/project_manager)
* Relaciones:

  * Un proyecto tiene muchas tareas.

**Tareas (`tasks`)**

* Campos: `title`, `description`, `status` (`pendiente`, `en_progreso`, `completada`), `project_id`, `assigned_to` (usuario asignado)
* Relaciones:

  * Una tarea pertenece a un proyecto.
  * Una tarea puede estar asignada a un usuario (developer).

---

#### 2Ô∏è‚É£ Relaciones resumidas

* `User (project_manager)` ‚Üí tiene muchos `Project`.
* `Project` ‚Üí tiene muchas `Task`.
* `Task` ‚Üí pertenece a `Project` y puede estar asignada a `User`.

---

#### 3Ô∏è‚É£ Endpoints principales (sin autenticaci√≥n)

| Endpoint        | M√©todo                 | Descripci√≥n       |
| --------------- | ---------------------- | ----------------- |
| `/api/users`    | GET, POST, PUT, DELETE | CRUD de usuarios  |
| `/api/projects` | GET, POST, PUT, DELETE | CRUD de proyectos |
| `/api/tasks`    | GET, POST, PUT, DELETE | CRUD de tareas    |

---

#### 4Ô∏è‚É£ Validaciones importantes

* No se puede eliminar un proyecto si tiene tareas asociadas.
* Solo se puede asignar tareas si existe el usuario y el proyecto.
* Validar campos obligatorios (`name`, `title`, `status`, etc.)

---

#### 5Ô∏è‚É£ Capa de servicios

Cada entidad tendr√° un **Service** que encapsula la l√≥gica de base de datos:

```
app/Services/
 ‚îú‚îÄ‚îÄ UserService.php
 ‚îú‚îÄ‚îÄ ProjectService.php
 ‚îî‚îÄ‚îÄ TaskService.php
```

Ejemplo de responsabilidades:

* **UserService**: crear, actualizar, eliminar, obtener usuarios.
* **ProjectService**: CRUD de proyectos, validaci√≥n de tareas antes de eliminar.
* **TaskService**: CRUD de tareas, asignaci√≥n a usuario, filtros opcionales.

---

#### 6Ô∏è‚É£ Flujo b√°sico

1. Controller recibe la request.
2. Controller pasa los datos al Service correspondiente.
3. Service valida y accede a la base de datos.
4. Controller retorna la respuesta JSON al cliente.

---

#### 7Ô∏è‚É£ Extras opcionales

* Filtros por estado de tareas: `/api/tasks?status=pendiente`
* Paginaci√≥n: `/api/projects?page=2&per_page=10`
* Preparar base para autenticaci√≥n y roles en el futuro.

## üîê 4. Crear endpoints de autenticaci√≥n

Creamos un **AuthController** para manejar login, registro y logout.

### üìÅ `app/Http/Controllers/AuthController.php`

```php
namespace App\Http\Controllers;

use Illuminate\Http\Request;
use Illuminate\Support\Facades\Hash;
use App\Models\User;

class AuthController extends Controller
{
    // üîπ Registro
    public function register(Request $request)
    {
        $validated = $request->validate([
            'name' => 'required|string',
            'email' => 'required|email|unique:users',
            'password' => 'required|min:6',
        ]);

        $user = User::create([
            'name' => $validated['name'],
            'email' => $validated['email'],
            'password' => $validated['password'],
            'role' => 'user', // Por defecto
        ]);

        return response()->json(['user' => $user], 201);
    }

    // üîπ Login
    public function login(Request $request)
    {
        $user = User::where('email', $request->email)->first();

        if (! $user || ! Hash::check($request->password, $user->password)) {
            return response()->json(['message' => 'Credenciales inv√°lidas'], 401);
        }

        $token = $user->createToken('api_token')->plainTextToken;

        return response()->json([
            'token' => $token,
            'user' => $user,
        ]);
    }

    // üîπ Logout
    public function logout(Request $request)
    {
        $request->user()->currentAccessToken()->delete();

        return response()->json(['message' => 'Sesi√≥n cerrada correctamente']);
    }
}
```

## üß± 6. Middleware de roles

### üìÅ `app/Http/Middleware/RoleMiddleware.php`

```php

use Closure;
use Illuminate\Http\Request;

class RoleMiddleware
{
    public function handle(Request $request, Closure $next, ...$roles)
    {
        $user = $request->user();

        if (! $user || ! in_array($user->role, $roles)) {
            return response()->json(['message' => 'No autorizado'], 403);
        }

        return $next($request);
    }
}

## üß≠ 7. Autorizaci√≥n con Policies

Las **Policies** controlan la autorizaci√≥n a nivel de modelo (qui√©n puede modificar, ver o eliminar algo).

### üìÅ `app/Policies/ProjectPolicy.php`

use App\Models\Project;

class ProjectPolicy
{
    public function viewAny(User $user): bool
    {
        return true;
    }

    public function view(User $user, Project $project): bool
    {
        return $user->id === $project->user_id || $user->role === 'admin';
    }

    public function update(User $user, Project $project): bool
    {
        return $user->id === $project->user_id || $user->role === 'admin';
    }

    public function delete(User $user, Project $project): bool
    {
        return $user->role === 'admin';
    }
}
```
